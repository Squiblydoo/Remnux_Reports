## 2. Classification

**Family:** Unattributed Node.js RAT with Ethereum on-chain C2 resolution
**Confidence:** HIGH (malicious) / MEDIUM (family attribution)
**Threat Type:** Remote Access Trojan / Loader / Crypto-targeting implant

**Reasoning:**
- Uses obfuscated variable names and multiple random `Start-Sleep` calls to evade static/behavioral analysis
- Hardcoded AES key in script decrypts two Node.js payloads
- Installs registry persistence (`HKCU\...\Run`) to survive reboots
- Main payload uses **on-chain C2 resolution** — queries public Ethereum RPC nodes to retrieve operator C2 URL from a smart contract (anti-takedown / anti-blocklist technique)
- Receives and executes arbitrary JavaScript code from C2 operator
- Self-reobfuscation capability prevents signature-based detection over time
- JavaScript obfuscation matches [javascript-obfuscator](https://github.com/javascript-obfuscator/javascript-obfuscator) tool (array rotation shuffle, `_0x` naming, Base64 string array)

## 3. Capabilities

### PowerShell Wrapper (ce.ps1)
- Suppresses all errors (`$ErrorActionPreference='SilentlyContinue'`)
- Generates **machine-fingerprinted hidden directory** path under `%LOCALAPPDATA%` using MD5(COMPUTERNAME+USERNAME), then maps to one of ~25 plausible Microsoft subdirectory paths (e.g., `Microsoft\Services\<8hex>`)
- Downloads **Node.js v18.17.0** from nodejs.org if not already present (3 retry attempts)
- Extracts Node.js ZIP to the hidden directory (`node-v18.17.0-win-x64\node.exe`)
- Decrypts two AES-CBC blobs and writes them as `<sha256hash>.idx` (main payload) and `<sha256hash>.db` (installer)
- Executes `node.exe <path_to_db>` with `-WindowStyle Hidden`
- State tracking: persists bot UUID and filenames as Base64 JSON to a short-named config file

### Blob 2 — Persistence Installer (`.db`)
- Reads state config file
- **Spawns** node.exe with `.idx` main payload (detached, hidden window, stdio=ignore)
- On first run only: sets **`HKCU\Software\Microsoft\Windows\CurrentVersion\Run`** → `node.exe <path_to_idx>` via `reg.exe` with `/f` flag
- Generates new bot tracking UUID on first run
- Uses `reg.exe add <key> /v <uuid> /t REG_SZ /d "<node.exe path_to_idx>"` syntax

### Blob 1 — Main RAT Payload (`.idx`)
#### On-Chain C2 Resolution
- Calls `eth_call` on multiple public Ethereum RPC endpoints (9 endpoints, queried in parallel with `Promise.allSettled`)
- Contract address and function selector hardcoded (obfuscated in string array)
- Decodes ABI-encoded `string` return value as the C2 URL (handles dynamic-type ABI encoding)
- URL must match `^(https?|wss?)://` to be accepted
- Votes across RPC responses for consensus URL → uses most-common result
- Refreshes every ~300 seconds (`0x493e0` ms)

#### Self-Update / Reobfuscation
- On startup: POSTs own source code to `<C2>/api/<botid>` along with build identifier
- If C2 returns ≥100-char response, writes it to disk (replacing itself) and re-launches via `spawn` then exits → achieves hot self-update with fresh obfuscation each time
- Prevents signature-based detection across runs

#### Main C2 Polling Loop
- Infinite loop: calls `_0x1d74b3()` then sleeps 500ms
- Constructs randomized polling URL: `<C2>/<4hex_prefix>/<botid>/<8hex_rand>.<ext>?<param>=<build>` where ext ∈ {`js`, `json`, `txt`, `css`, `html`, `xml`} and param ∈ {`id`, `ref`, `type`, `b`, `q`, `s`, `v`} — mimics legitimate CDN/static asset requests
- Sends `X-Bot-Server: <c2_url>` header (control signal to C2)
- 120-second timeout (`AbortController`)
- If response body >10 chars: executes as JavaScript via `new AsyncFunction('require','process','Buffer','console','__dirname','__filename','log', <payload>)` — equivalent to eval with full Node.js API access

#### Debug Logging
- Writes timestamped log to `%APPDATA%\<obfuscated_path>` when debug flag enabled (stored in state config field `5`)

## 4. Attack Chain

```
ce.ps1 (PowerShell loader)
  │
  ├─ [1] Generate hidden %LOCALAPPDATA% directory (MD5-derived, mimics Microsoft paths)
  ├─ [2] Download node.exe v18.17.0 from nodejs.org if absent
  ├─ [3] Decrypt two AES-CBC blobs → write <hash>.idx + <hash>.db
  └─ [4] Execute: node.exe <hash>.db (hidden)
           │
           ├─ [5] Spawn: node.exe <hash>.idx (detached, hidden) → Main RAT
           └─ [6] Set HKCU\...\Run persistence (first run only)
                    │
                    └─ node.exe <hash>.idx on every Windows logon
                        │
                        ├─ [7] Query 9 Ethereum RPC nodes → decode C2 URL from smart contract
                        ├─ [8] POST self to C2 /api/<botid> → receive re-obfuscated version, restart
                        └─ [9] Poll C2 with randomized URLs every ~500ms
                                 └─ [10] Execute operator JS payload via AsyncFunction
```

## 6. Analyst Notes

### Gaps Requiring Dynamic Analysis
1. **Ethereum smart contract address**: fully obfuscated, not statically recoverable without executing the string decoder. Running the JS in an isolated Node.js environment would reveal the contract address and current C2 URL.
2. **5 additional Ethereum RPC URLs**: also obfuscated in string array. Dynamic execution would reveal all 9 endpoints.
3. **Actual C2 server**: depends on smart contract state at execution time. The contract could be updated by the operator at any time (resilient C2 infrastructure).
4. **Final operator payload**: this is a second-stage loader — the actual malicious activity (credential theft, cryptostealing, etc.) is delivered dynamically by the C2 and never touches disk in the initial sample.
5. **Build/version identifier** (`_0x30ac57`): obfuscated — likely a campaign tag.

### Notable Techniques
- **On-chain C2 resolution** is a sophisticated anti-takedown technique: no traditional domain to sinkhole, requires Ethereum blockchain access to block. Seen in several 2024-2025 crypto-targeting campaigns.
- **Living-off-the-land with Node.js**: uses the victim's (or freshly downloaded) Node.js runtime rather than a compiled PE, making behavioral detection harder.
- **Self-reobfuscation**: operator can push fresh obfuscation at will, defeating signature-based EDRs.
- **Directory camouflage**: the MD5-derived path maps to plausible `Microsoft\Services\` style directories, blending with legitimate application data.
- **Randomized sleep jitter** throughout PS1: `Start-Sleep -Milliseconds (Get-Random -Min X -Max Y)` on nearly every operation — likely to evade sandbox timing heuristics.
- **Three-attempt retry** on Node.js download with file-size validation (>1MB).

### Delivery Vector
Unknown — `ce.ps1` is likely delivered via:
- Malicious npm package (`.js` → spawns PowerShell)
- Phishing / malvertising (PowerShell execution via `cmd /c powershell -ep Bypass`)
- Trojanized software installer (similar to other samples in this workspace)

### Alternative Hypotheses
- The `localhost:3000` C2 fallback and JavaScript obfuscator tooling are consistent with a developer/pentester who open-sourced a RAT framework — this could be a commercial or open-source RAT adapted for crypto theft.
- Node.js v18.17.0 specifically (older LTS, not the latest) may indicate the threat actor tested against a specific version.
