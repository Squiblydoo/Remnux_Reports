## Executive Summary

`USDC_Case_47293-A3_Notice.exe` is a **SimpleHelp Remote Access Client v5.2.11**
weaponised using the **Authenticode stuffing** technique described in the
[GData blog post (June 2025)](https://blog.gdatasoftware.com/2025/06/38218-connectwise-abuse-malware).
The attacker took the legitimate, signed SimpleHelp binary, left the PE code
untouched (preserving the valid Authenticode signature), and embedded a custom
server configuration pointing to a recently-created attacker-controlled domain
in an unauthenticated PKCS#7 attribute — a region of the file that is
**explicitly excluded from the Authenticode hash calculation**.

The lure name ("USDC Case") mimics a U.S. District Court notice to coerce
execution. On launch, the binary silently installs a fully-functional remote
access agent connecting to `online.healthcare-shore.com:443`.

## Technique: Authenticode Stuffing

### How Authenticode Works

When Windows verifies an Authenticode signature it computes an **AuthentiHash**
over the PE file, deliberately skipping three regions:

1. The `Checksum` field in the Optional Header
2. The `Certificate Table` entry (RVA + size) in the Data Directory
3. The entire **WIN_CERTIFICATE** block (the certificate table itself)

The certificate table holds a PKCS#7 `SignedData` structure.  Within
`SignedData`, the `SignerInfo` element contains two attribute sets:

| Set | Covered by hash? |
|-----|-----------------|
| **authenticatedAttributes** (`signedAttrs`) | **Yes** — included in the digest |
| **unauthenticatedAttributes** (`unsignedAttrs`) | **No** — never hashed |

Anything placed in the unauthenticated attributes can be freely added,
modified, or replaced without touching the bytes that Windows validates.

### This File

The certificate table starts at file offset **0x1712F0** (209,752 bytes,
running to EOF). The PE sections end at **0x65000**, leaving a 1,098,480-byte
region of SimpleHelp's own binary data before the cert.

Inside the PKCS#7, the `SignerInfo.unsignedAttrs` block is **205,630 bytes**
— vastly larger than a normal countersignature — and contains two attributes:

| # | OID | Size | Content |
|---|-----|------|---------|
| 0 | `1.2.840.113549.1.9.6` (countersignature) | 670 bytes | GlobalSign timestamp |
| 1 | `00.00.00.00.00.00` (non-standard) | **204,956 bytes** | SimpleHelp configuration |

Attribute 1 uses the **all-zero OID** `00 00 00 00 00 00` — this is the key
reason the GData script did not work: it searches for a specific known OID
(or uses `asn1crypto` which expects a registered OID). This file's
non-standard OID bypasses that check.

### Why the GData Script Failed

The GData script enumerates unauthenticated attributes by matching known OIDs
or relies on `asn1crypto`'s OID registry. An all-zero OID (`0.0.0.0.0`) is
unregistered and causes the parser to either skip the attribute, raise an
exception, or return no match — resulting in zero output despite the
configuration being present.

| **Validity** | 2018-01-16 → **2021-03-16** (expired) |
| **Timestamp** | **2020-06-03 10:50:50 UTC** (GlobalSign, within validity) |

### Why the Signature Appears Valid Despite an Expired Certificate

Windows (`WinVerifyTrust`) honours **countersignature timestamps**:

> If a file was signed **and timestamped** during the certificate's validity
> window, the signature is considered valid indefinitely — even after the
> signing certificate expires.

The SimpleHelp binary was originally signed in June 2020 (within the
Jan 2018 – Mar 2021 window) with a GlobalSign RFC 3161 timestamp.
The threat actor:

1. **Left the PE code completely intact** — the AuthentiHash is unchanged.
2. **Left the original `SignedData` structure intact** — same cert, same digest.
3. **Only modified the unauthenticated attributes** — adding attribute 1 with
   their own server configuration.

Because the unauthenticated attributes are not hashed, Windows still
computes the same AuthentiHash, finds it matches the signed digest, and
reports the signature as **valid**. The certificate displayed to the user
shows *"SimpleHelp Ltd" / Thawte* — not the attacker's identity.

The recently-created domain `healthcare-shore.com` is **not part of
certificate validation** — Windows never checks what server the software
calls home to.

SimpleHelp v5.2.11 provides the attacker with a full-featured RAT:

- Remote desktop / screen capture
- Remote shell / terminal

- Keyboard and mouse injection
- Clipboard access

- Silent background operation (`show_no_ui = true`)

## Detection Notes

- Standard AV will recognise SimpleHelp as a legitimate RMM tool and may not
  flag it; focus detection on the domain `healthcare-shore.com` and
  the non-standard all-zero OID in the PKCS#7 unauthenticated attributes.
- YARA: search for `sg_silent_install` + `sg_servers` within
  `0x1712F0 + 8` offset (inside the cert table).
- Network: block `online.healthcare-shore.com` and alert on SimpleHelp
  UDP/443 beaconing to non-corporate domains.
