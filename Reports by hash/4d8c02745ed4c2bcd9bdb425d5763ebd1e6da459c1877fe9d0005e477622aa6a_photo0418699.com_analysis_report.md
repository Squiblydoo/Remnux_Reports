**Reasoning**: The binary is a PE executable disguised with a `.com` extension (a common Windows execution trick). It employs a multi-stage custom packer with LZX decompression, XOR decryption, and control flow obfuscation. The fully-resolved API set exposes a WinHTTP C2 client, triple-method anti-debug, process injection, system fingerprinting, and file-drop capabilities. An EV certificate issued to a Xiamen, China entity is consistent with the short-lived cert pattern used by CN-linked threat actors.

## 3. Capabilities

### Packer / Loader Stage
- **LZX decompression**: Full LZX decompressor (Huffman + suffix array sort) in the `.text` stub decrypts the packed `.textbss` virtual section (995,328 bytes total)
- **In-place section decryption**: VirtualProtect(RWX) → decrypt → VirtualProtect(original perms) per-section; 7 sections decrypted in order
- **XOR-in-loop obfuscation**: 15 XOR loop instances detected; used for both payload decryption and anti-analysis
- **Anti-disassembly**: LOCK/UNLOCK x86 prefix sequences inserted before indirect jumps to defeat linear-sweep disassemblers; unrecoverable jumptables in entry path
- **TLS callbacks**: TLS directory present — anti-debug code runs **before** the entry point
- **Dynamic import by name**: Resolves all payload imports at runtime via GetProcAddress loops (×232 calls), preventing static IAT analysis
- **ImportByHash capability**: capa detected "resolve function by parsing PE exports" — hash-based API lookup used in the packer stub itself

### Payload Stage (resolved from emulation)
**C2 Communication**
- Full WinHTTP stack: `WinHttpOpen`, `WinHttpConnect`, `WinHttpCrackUrl`, `WinHttpOpenRequest`, `WinHttpSendRequest`, `WinHttpReceiveResponse`, `WinHttpReadData`, `WinHttpQueryHeaders`, `WinHttpCloseHandle`, `WinHttpSetOption`

**Anti-Analysis (3 methods)**
- `IsDebuggerPresent` — basic debugger flag check
- `CheckRemoteDebuggerPresent` — remote debugger detection
- `NtQueryInformationProcess` — debugger and process information check
- `GetTickCount64` — timing-based sandbox detection
- `timeSetEvent` / `timeKillEvent` — high-resolution timer for timing checks
- `AddVectoredExceptionHandler` / `RemoveVectoredExceptionHandler` — VEH abuse (anti-debug or hook installation)

**System Fingerprinting**
- `EnumSystemFirmwareTables` + `GetSystemFirmwareTable` — UEFI/BIOS firmware table enumeration for hardware-based fingerprinting (used to detect VMs with no firmware tables)
- `NtQuerySystemInformation` — kernel-level system information
- `GetLocalTime`, `QueryPerformanceCounter/Frequency` — timing and locale

**Process Injection / Memory Manipulation**
- `NtSetContextThread`, `NtGetContextThread`, `NtOpenThread` — thread context read/write for process injection
- `NtSetInformationThread` — HideThreadFromDebugger and other thread manipulation
- `NtAllocateVirtualMemory`, `NtFreeVirtualMemory` — direct NT memory alloc (bypasses user-mode hooks)
- `GetThreadContext` — injection context capture
- `VirtualProtect` — permission manipulation

**Registry / AV Detection**
- `RegOpenKeyExA`, `RegQueryValueExA`, `RegEnumKeyExA`, `RegQueryInfoKeyA` — registry enumeration (likely scanning for installed AV/sandbox products)
- `StrStrIA` — case-insensitive substring search (likely for process/AV name matching against a hardcoded list)

**File Operations**
- `GetTempPathA`, `CreateFileA/W`, `WriteFile`, `DeleteFileA`, `SetFileAttributesA`
- `FindFirstFileExW`, `FindNextFileW`, `FindClose`
- `CreateDirectoryA`, `SetFilePointerEx`

**Execution**
- `ShellExecuteA` — used to launch dropped/downloaded payloads
- `GetCommandLineA/W` — command line inspection

**Path Resolution**
- `SHGetFolderPathA` — resolve special folders (AppData, Temp, Windows dir)
- `PathCombineA`, `GetWindowsDirectoryA`

## 6. Emulation Results

**Tool**: speakeasy 1.5.11 + custom runner hooks  
**Architecture**: amd64  
**Runtime**: 42.7 seconds  
**API calls**: 845 (mostly HeapAlloc×288, GetProcAddress×232, HeapFree×158)

**Key findings**:
- Packer makes 14 VirtualProtect(RWX) calls to decrypt 7 sections in-place
- Payload code begins executing from VA ~0x14004a000 after unpacking
- All payload imports resolved dynamically (WinHTTP, NT functions, CRT, etc.)
- Emulation terminated at unsupported `kernel32.NtAllocateVirtualMemory` call
- No network events, registry writes, or mutex creation captured before termination
- Memory dumps of all 7 decrypted sections captured (contents remain encrypted at code level — a second decryption pass occurs at runtime for individual strings/URLs)

**Memory dumps**: 7 RWX regions at:
- `0x140001000` (188KB), `0x14002f000` (81KB), `0x140043000` (12KB)
- `0x140046000` (12KB), `0x140049000` (4KB), `0x14004a000` (663KB), `0x1400ec000` (32KB)
- Combined: 995,328 bytes = exact `.textbss` virtual section size

**Dynamic binary string** (ea 0x14018408D): `D6B5C0ADEE27EA604FEBD4273DAEB2C200000000000000007935147A4E86C861` — 32-byte runtime-constructed blob, possible encryption key or C2 token.

## 9. Analyst Notes

**Residual gaps**:
1. **C2 URL not recovered** — payload performs a second runtime string decryption pass for URLs/hostnames after `NtAllocateVirtualMemory`. Dynamic execution in a real sandbox (Tria.ge, Any.run, Cuckoo) is needed.
2. **Payload family unknown** — the unpacked code size (~995KB) and capability set (WinHTTP + injection + fingerprinting) are consistent with an information stealer or RAT, but no family signature matched in YARA or capa.
3. **Dynamic binary string** `D6B5C0ADEE27EA60...` may be decryptable — could be a campaign ID, AES key, or hardcoded token. angr symbolic execution on the construction routine could recover this.

**Naming pattern note**: The `photo<digits>.com` filename pattern is superficially similar to `photo20260411689.com` (previously analyzed Zhong Stealer). However, no cross-reference indicators (cert serial, C2 IOC, config values, PDB path) were confirmed matching. The two samples should be treated independently until concrete overlap is established.

**Recommended follow-up**:
1. Submit to Tria.ge or Any.run for dynamic C2 URL extraction
2. Patch speakeasy to stub `NtAllocateVirtualMemory` and continue emulation past the breakpoint
3. Use angr on the dynamic string construction function to extract the 32-byte blob origin
4. Search for other samples signed by cert `66220da9448f87f52007149619ea4b37` (Xiamen Xianghe) on VirusTotal
