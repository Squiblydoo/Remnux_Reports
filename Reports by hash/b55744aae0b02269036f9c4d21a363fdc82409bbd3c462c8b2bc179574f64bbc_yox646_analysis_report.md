## 2. Classification

**Family:** Unknown — novel or untracked trojanized text-editor dropper  
**Confidence:** HIGH CONFIDENCE MALICIOUS  
**Reasoning:**  
The file presents as a functional text editor (lure: window class names `W3.Main` / `W3.Aux` from Notepad3, full drag-and-drop, open/save/edit UI), but contains a hidden payload decryption and execution path gated behind window-metrics key derivation and a module presence check. No known family signatures matched (KesaKode: no verdict; YARA: MSVC compiler only). The Sectigo EV cert belongs to a 1-year-old Fujian/Xiamen entity consistent with fraudulently-obtained Chinese EV infrastructure seen across multiple unrelated campaigns.

---

## 3. Capabilities

- **Functional text-editor lure** — implements drag-and-drop file open, open/save dialogs, EDIT control, LISTBOX, font chooser, timer-based ticker, all consistent with Notepad3 (W3 window class)
- **Window-metrics key derivation** — `sub_1400013c0` computes a 64-bit hash from 8 `uint16_t` font/window metrics values using magic constant `0x7152747bf86aa000`, storing the result at `[0x14002a528]`; this key is unique per system/rendering environment, defeating sandboxes
- **Runtime payload decryption** — `sub_1400010b0` subtracts the derived hash from two pairs of encrypted global variables to reveal valid function pointers (`GetProcAddress`/`VirtualAlloc` equivalents), then decrypts ~3,062 bytes of payload from `.data` (VA range ~`0x1400228xx`–`0x1400298c8`) using an additive running-key cipher (keys start 0/−1 and decrement by 2 per step, combined with a phase term `(uVar6/5 & 0xFE)*5`)
- **Conditional execution gating** — `GetModuleHandleA([0x140029940])` checks for presence of an obfuscated module name; decryption and payload execution only proceed if the module is loaded
- **Shellcode execution** — allocates a buffer (`~0x1000` bytes via dynamically resolved `VirtualAlloc`-equivalent), writes decrypted code whose first byte is `0xE9` (x86 JMP opcode), then calls the buffer via indirect function pointer
- **Full string obfuscation** — all sensitive string literals are constructed dynamically on the stack via SIMD-like 49-byte array initialization blocks; no plaintext module names, API names, or C2 indicators appear in the binary (16 `StackArrayInitialisationX64` instances, 14 `DynamicString` instances)
- **Runtime API resolution** — dynamically resolves function pointers with encrypted/rotated pointer storage (pointer XOR/rotate with `SecurityCookie`); capa confirms "link function at runtime on Windows" (T1129)
- **Cross-section JMP** — anomaly at EA 1599 (early `.text`); control flow jumps across sections, consistent with code patching or positional payload placement
- **File discovery** — reads and writes files (capa T1083/T1082)

## 4. Attack Chain

```
Stage 1 — Social Engineering
  User executes yox646.exe (appears to be a text editor)
      ↓
Stage 2 — Lure Initialization (WM_CREATE)
  Main WndProc (sub_140001cc0) creates full-featured editor UI
  Sets 2-second WM_TIMER to tick text updates
  Calls sub_1400013c0 to create secondary window
      ↓
Stage 3 — Key Derivation (sub_1400013c0)
  Creates BUTTON3/BUTTON4/BUTTON5 UI buttons
  Calls sub_140007b90() to get 8 uint16_t font/window metrics values
  Computes 64-bit hash: H = (metric[0]^2 + 0x7152747bf86aa000)
                           accumulating metric[k]*2^k contributions
  Stores H in [0x14002a528]; adjusts [0x140029920] -= H
  Calls sub_1400010b0()
      ↓
Stage 4 — Conditional Payload Activation (sub_1400010b0)
  GetModuleHandleA([0x140029940])  ← obfuscated module name
  IF module not loaded → exit (evasion gate)
  [0x1400298e0] -= H  (decrypt global function pointer #1)
  [0x1400298e8] -= H  (decrypt global function pointer #2)
  [0x140029900] -= H  (decrypt global function pointer #3)
  [0x140029908] -= H  (decrypt global function pointer #4)
  GetProcAddress(module, [0x140029930]) → allocator function
  VirtualAlloc(0, ~4032, MEM_COMMIT, PAGE_EXECUTE_READWRITE)
      ↓
Stage 5 — Payload Decryption
  Additive running-key cipher decrypts 3,062 bytes
  Source: .data[0x1400298c8 .. 0x14002a4xx] (stored reversed)
  Keys: cVar8 = {0,−2,−4,...}, cVar4 = {−1,−3,−5,...}, phase = (n/5 & 0xFE)*5
  Output buffer starts with 0xE9 (JMP) → shellcode
      ↓
Stage 6 — Shellcode Execution
  (*decrypted_buffer)(buffer, size, base_addr, params)
  Stage-6+ IOCs encrypted within shellcode (not recovered)
```

## 8. Analyst Notes

**Key gaps:**
1. **Module name unknown** — the string at `[0x140029940]` passed to `GetModuleHandleA` is obfuscated via stack array construction; reversing it requires a live execution trace or manual recreation of the obfuscation scheme. Candidate: `ntdll.dll`, `user32.dll`, or a third-party DLL indicating a staged infection (module only present after prior-stage malware has run).
2. **Stage-6 payload unknown** — the ~3KB shellcode beginning with `E9` (JMP) could be a downloader, in-memory loader, keylogger injector, or any second-stage; C2 domains/IPs, encryption keys, and behavioral profile are all deferred to that payload.
3. **Trigger timing** — it is unclear exactly when `sub_1400013c0` is called. Given that a 2-second timer is set in WM_CREATE and the secondary window functions are also created at startup, the decryption could run immediately on first paint or be deferred to a user action.

**Recommended follow-up:**
- **Live execution trace** (Frida or WinDbg) with `GetModuleHandleA` and `VirtualAlloc` hooks to capture the decrypted shellcode buffer
- **Patch the module check** (NOP the `cmp rax, 0` conditional jump at the module-not-found exit) and run in a VM to force payload execution
- **Extract and carve** the `.data` block at VA `0x140028xx`–`0x1400298c8` (raw bytes at file offsets ~160988–167112), write a standalone decryptor using the additive cipher with known-zero initial keys (or brute-force the 64-bit hash if font metrics can be enumerated for common Windows configs)
- **Threat intel pivoting** on certificate serial `0082faaa1d55081c3ded58ae3311d04f55` and the parent company "Xiamen Chike Network Technology" — fresh 1-year EV cert from a Fujian/Xiamen entity warrants checking for sibling samples

**False positive considerations:**
- The text editor functionality is real and complete; the binary could theoretically be a legitimate Notepad3-based tool with all suspicious indicators being false positives. However, the combination of: (1) a self-modifying pointer table, (2) ~3KB of encrypted code payload, (3) a conditional execution gate, (4) complete string obfuscation, and (5) 0/100 sandbox score despite behavioral analysis is not consistent with legitimate software. **Confidence: HIGH MALICIOUS.**
