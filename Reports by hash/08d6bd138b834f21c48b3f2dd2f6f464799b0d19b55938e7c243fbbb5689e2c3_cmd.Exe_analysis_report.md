## 2. Classification

**MALICIOUS — HIGH CONFIDENCE (re-signing / supply chain artifact)**  
**Family:** Re-signed legitimate Windows binary (Authenticode fraud)  
**TTPs:** T1036.001 (Masquerading: Invalid Code Signature), T1554 (Compromise Host Software Binary), T1553.002 (Subvert Trust Controls: Code Signing)

**Revised Reasoning:** Dynamic analysis (speakeasy emulation) and decompilation of all flagged functions confirm this binary contains **no injected malicious code**. It is a genuine, unmodified copy of Windows cmd.exe (build 10.0.17763.1697) from which the legitimate Microsoft Authenticode signature was stripped and replaced with a fraudulent EV certificate from a Chinese technology company. The threat is the re-signing act itself, not any embedded payload.

## 3. Initial Analysis Errors (Corrected)

The initial report (2026-04-05) incorrectly attributed several behaviors as malicious. All were false positives:

| Initial Claim | Correction |
|---|---|
| DEP disable via `NtSetInformationProcess(ProcessExecuteFlags=0x27, 2)` | Class 0x27 = **ProcessPagePriority** (not ProcessExecuteFlags). cmd.exe's COPY command sets page priority to MEMORY_PRIORITY_BELOW_NORMAL during file operations and restores it afterward. Standard Windows optimization. |
| Reflective PE loader (`sub_4135fc`, ~62KB) | Decompile confirms: `sub_4135fc` is the **COPY command handler** — calls `CopyFileW`, `ReadFile`, `memmove`. Its size reflects COPY's many code paths (/A, /B, concatenation, etc.). |
| VirtualAlloc for shellcode (`sub_415568`) | Decompile confirms: memory allocator for COPY operation's I/O buffers. Allocates `PAGE_READWRITE` (not RWX), has low-memory retry logic, error path calls `ERROR_NOT_ENOUGH_MEMORY` (code 8). |
| 16 XOR-in-loop decryption functions | All 16 are the **MSVC stack cookie verification pattern**: `XOR ECX, EBP` (opcode `33 CD`) followed by `CALL __security_check_cookie`. Not decryption loops. |
| Dynamic VirtualProtect resolution | `VirtualProtect` string present because cmd.exe's ASSOC/FTYPE implementation processes API names as text; confirmed not in IAT and not resolved at runtime. |
| Stage-2 payload (XOR-encrypted) | No payload exists. Speakeasy emulation captured zero allocations, zero DEP events, zero thread starts across entry point run and direct function calls. |
| PE export resolver | Confirmed to be COPY command I/O implementation. |

## 4. Actual Behavior

The binary executes as an **unmodified Windows cmd.exe** in every respect. Behavioral analysis would show normal command interpreter activity with no injected code path.

The only anomaly observable at runtime:
- Authenticode certificate chain verification will fail (not signed by Microsoft)
- File hash does not match known-good Windows cmd.exe (due to signature blob replacement)

### What the Binary Actually Does
- Full Windows command interpreter: FOR, IF, GOTO, CALL, SET, COPY, etc.
- `NtSetInformationProcess(0x27)` in COPY path: sets page priority to below-normal for large file operations (legitimate Windows optimization)
- `VirtualAlloc(MEM_COMMIT, PAGE_READWRITE)`: COPY I/O buffer allocation with low-memory retry
- `GetProcAddress("RaiseFailFastException")` from kernelbase: legitimate error handling

## 5. Threat Assessment

### Threat: Fraudulent Re-Signing

This artifact represents a **re-signed supply chain / filesystem hijack component**. Likely use cases:

1. **cmd.exe replacement (T1554)**: Dropped into a target environment as a replacement for `C:\Windows\System32\cmd.exe`. Because the code is identical to the legitimate binary, behavior-based detections do not trigger. File-hash detections may miss it if signatures rely on known-bad hashes rather than known-good hashes.

2. **Whitelisting / SmartScreen bypass (T1553.002)**: The EV certificate provides SmartScreen reputation bypass on delivery. Environments with lax application whitelisting (any valid EV cert accepted) would execute this file without warning.

3. **Integrity monitoring evasion (T1036.001)**: File integrity monitoring (FIM) tools comparing against known-good SHA256 hashes would alert; but tools checking only for "valid digital signature" may not alert on the certificate substitution.

### What This Is Not
- There is no embedded payload
- There is no stage-2 loader
- There is no C2 communication
- There is no DEP manipulation
- There is no shellcode or reflective loading

## 7. Detection Guidance

### File Integrity Monitoring
Compare against known-good SHA256 for cmd.exe build 10.0.17763.1697. Any mismatch on a system binary should alert regardless of signature validity.

### Authenticode Validation
Verify the signing certificate chain anchors to Microsoft's root CA. Any Windows system binary signed by a non-Microsoft certificate is anomalous.

### YARA
```yara
rule ResignedWindowsBinary_XiamenJiaming {
    meta:
        description = "Windows system binary re-signed with Xiamen Jiaming EV cert"
        sha256 = "08d6bd138b834f21c48b3f2dd2f6f464799b0d19b55938e7c243fbbb5689e2c3"
        reference = "cmd.Exe — re-signed Windows 10 1809 cmd.exe"
    strings:
        $ms_claim    = "Microsoft Corporation" wide
        $jiaming_cn  = "Xiamen Jiaming Network Technology" ascii
        $serial      = { 00 AC B4 AF 64 AC 06 1A AD B0 0B 53 70 B4 CA 02 46 }
    condition:
        uint16(0) == 0x5A4D and $ms_claim and ($jiaming_cn or $serial)
}

rule SuspiciousWindowsBinaryResigning {
    meta:
        description = "Windows system binary (VersionInfo=Microsoft) signed by non-Microsoft cert"
    strings:
        $ms_corp    = "Microsoft Corporation" wide
        $ms_windows = "Microsoft" wide
        $ms_root    = "Microsoft Root" ascii
    condition:
        uint16(0) == 0x5A4D and
        ($ms_corp or $ms_windows) and
        not $ms_root
}
```

## 8. Threat Actor Notes

- The cert was issued 2026-03-24 — 12 days before analysis. Fresh EV cert issuance is consistent with active campaign infrastructure.
- The Sectigo EV R36 CA chain has appeared across multiple samples in this workspace (sethc.exe, Wax Setup.msi). This CA is being systematically abused for EV cert fraud in this campaign cluster.
- Xiamen (Fujian Province, CN) mirrors the Xiamen Xisu Technology cert seen in NovaViewer.exe (2026-03-10), though the entities differ. This may indicate a cert procurement service or common threat actor methodology in the region.
- The capital-E `cmd.Exe` filename is a deliberate artefact — Windows filenames are case-insensitive, so `cmd.Exe` replaces `cmd.exe` transparently on Windows NTFS. The deviation may help the attacker track which systems received the substitution.

## 9. Analyst Notes

### Summary
This is not a functioning malware implant. It is a legitimate Windows binary that has been re-signed to create a plausible-looking replacement for `cmd.exe`. Its value to an attacker is **positional** — being placed where the real `cmd.exe` would normally be — rather than functional. The binary itself will not do anything a legitimate cmd.exe would not do.

### Gaps
- The source of distribution (how this binary is dropped onto target systems) is unknown — no dropper was analyzed.
- Whether this was observed actively replacing `cmd.exe` in a victim environment is unknown.
- The specific whitelisting policy being targeted (if any) is unknown.
