## 2. Classification

**MALICIOUS — HIGH CONFIDENCE**  
**Family:** Trojanized cmd.exe / Reflective PE Loader  
**TTPs:** T1036.001 (Masquerading: Invalid Code Signature), T1055 (Process Injection), T1140 (Deobfuscate/Decode), T1546.001 (Persistence: Change Default File Association)

**Reasoning:** The binary preserves genuine Windows cmd.exe functionality (version 10.0.17763.1697 = Windows Server 2019 / Windows 10 1809) as a lure, while embedding a reflective PE loader that:
1. Temporarily disables DEP/NX
2. Resolves PE exports via custom export walker
3. Allocates RW memory for payload execution
4. Dynamically resolves `VirtualProtect` without importing it

The identity fraud (Microsoft VersionInfo + Chinese company EV cert) is definitively malicious. No legitimate use case exists for signing a Microsoft system binary with a third-party certificate.

## 3. Capabilities

### Confirmed Malicious Behavior

- **DEP/NX Disabling** — `sub_4155ad` calls `NtSetInformationProcess(-1, ProcessExecuteFlags=0x27, 2, 4)` to disable Data Execution Prevention for the process before payload execution; restores original state afterwards (DEP save → disable → execute → restore pattern)
- **Reflective PE Loader** — `sub_4135fc` (~62KB, most complex function in binary) walks PE export directories to resolve function addresses; called from within the DEP-disable context
- **Memory Allocation for Shellcode** — `sub_415568` calls `VirtualAlloc(NULL, size, MEM_COMMIT=0x1000, PAGE_READWRITE=4)`; RW memory becomes executable after DEP disable; allocated buffers stored in indexed global array at `0x43576c`
- **Dynamic API Resolution** — `VirtualProtect` present as string in ImportNames but **not in IAT** (confirmed no import entry); resolved at runtime via GetProcAddress (flagged by malcat `PossiblePackerApiDynamicImport`)
- **XOR Obfuscation** — 16 functions contain XOR-in-loop constructs (obfuscated strings or payload decryption)
- **PEB Walking** — `code::PEBx86` constant detected; used for position-independent API resolution
- **Anti-Debug** — `IsDebuggerPresent`, `OutputDebugStringW`, `UnhandledExceptionFilter`, `TerminateProcess`; stack-based dynamic string construction (2 instances)

### Legitimate cmd.exe Features (Cover)
- Full command interpreter (FOR, IF, GOTO, CALL, SET, etc.)
- Registry read/write via ASSOC/FTYPE commands
- Process creation via START command (VirtualAlloc/ResumeThread via delay imports)
- Directory operations (MKDIR, RMDIR, PUSHD, POPD, MKLINK)
- PE section enumeration for .NET assembly detection
- GetProcAddress usage limited to `RaiseFailFastException` from `kernelbase.dll` (legitimate)

## 4. Attack Chain (Reconstructed)

```
[System] ──► cmd.exe replaced with trojanized version
                │
                ▼
[User invokes cmd.exe]
                │
                ├─► [Normal cmd.exe functionality runs] ← cover / user trust
                │
                └─► [sub_4155ad: Loader routine]
                         │
                         ├─ 1. NtQueryInformationProcess(ProcessExecuteFlags) → save old DEP state
                         ├─ 2. NtSetInformationProcess(ProcessExecuteFlags, 0x2) → DISABLE DEP
                         ├─ 3. sub_4135fc: PE export resolver (reflective loader core)
                         │       └─ Walks PE export directory tables
                         │       └─ Resolves function addresses by export name
                         ├─ 4. sub_415568: VirtualAlloc(NULL, size, MEM_COMMIT, PAGE_READWRITE)
                         │       └─ Allocates RW memory → executable because DEP is disabled
                         │       └─ Stores pointer in global array [0x43576c + n*4]
                         ├─ 5. [Payload execution — stage 2 not recovered in static analysis]
                         └─ 6. NtSetInformationProcess(ProcessExecuteFlags, saved) → restore DEP
```

**Stage 2 not recovered**: The reflective loader payload is not present in cleartext — likely XOR-decrypted at runtime by one of the 16 XOR-in-loop functions. Dynamic analysis required to recover the payload.

## 6. Analyst Notes

### What's Confirmed
1. **Identity fraud is definitive** — no legitimate scenario for a Chinese company EV cert on a binary claiming to be Microsoft's `cmd.exe`
2. **DEP-disable + reflective loader is definitively malicious** — this behavior does not exist in any version of Microsoft's cmd.exe and serves no legitimate purpose for a command interpreter
3. **The cmd.exe functionality is real** — the binary contains genuine Windows cmd.exe code (10.0.17763.1697), likely taken directly from a Windows Server 2019 / Windows 10 1809 install and patched with the loader code

### What Requires Dynamic Analysis
- **Stage 2 payload not recovered** — the reflective loader's target is XOR-obfuscated; executing under a hypervisor/sandbox would reveal what's loaded
- **Trigger conditions unknown** — geographic check (T1614) suggests the loader may only activate for specific regions/languages; anti-debugging may suppress it in sandboxes
- **Persistence mechanism scope** — the file association registry capability was flagged but it's unclear if it's used maliciously (ASSOC/FTYPE cover) or as a secondary persistence mechanism

### Hypotheses
1. **Primary hypothesis**: This is a supply-chain or spear-phishing artifact that replaces `cmd.exe` in a target environment. The trojanized binary runs normally to avoid detection while loading a second-stage payload (backdoor, RAT, or credential stealer) into the cmd.exe process space.
2. **Alternative**: This is a test/staging build; the loader may be functional but the XOR-encrypted payload may not be embedded — requiring it to be fetched from a C2 (not observed in static analysis).

### Threat Actor Notes
- The EV cert from Xiamen (Fujian Province, CN) mirrors the Xiamen Xisu Technology cert seen in NovaViewer.exe (analyzed 2026-03-10), though the entities differ. This may indicate a recurring pattern of Chinese EV cert abuse by the same threat actor cluster or a cert-procurement service operating in the region.
- The cert was issued 2026-03-24 — fresh issuance, 12 days before analysis — consistent with active campaign infrastructure.
- The Sectigo EV cert chain (CA EV R36) has appeared in multiple recent samples in this workspace (sethc.exe, Wax Setup.msi). This specific CA appears targeted for EV cert fraud.
